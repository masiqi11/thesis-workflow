# Thesis Workflow

## 仓库定位

这是一个面向 Claude Code 的 GitHub-ready Skill 仓库。

其目标不是把论文写作变成“黑盒一键生成”，而是把高风险长链任务拆成：
- intake 闸门
- 证据收集
- 提纲与资产规划
- 章节写作
- 内容核对
- 引用真值核验
- 格式审查
- AI 风险审查
- 降重优化
- 终稿构建

若将本仓库公开发布，应把它理解为：**workflow + prompt assets + helper tools + examples**。

## 总则

这是**通用型论文写作 Skill**，不是针对某一学校模板硬编码的固定脚本。

### 第一铁律
**未澄清要求，不得开始正式论文编写。**

在以下信息未明确前，本 Skill 不得直接进入正式正文撰写：
- 学校/学院/专业要求
- 论文结构要求
- 标题编号规则
- 字体字号
- 行距段距
- 页边距
- 页码规则
- 图题表题规则
- 公式规则
- 参考文献格式
- 摘要/Abstract/致谢/附录要求

### 模板限制说明
我无法仅凭一个 `doc/docx` 模板文件，稳定、完整、准确地提取全部学校排版细则。
若模板中的格式要求对最终稿有约束，必须由用户提供**文字版格式要求**，或将模板要求整理成可执行条目。

### 仓库配套资产
本仓库建议同时携带以下配套内容：
- `README.md` / `README_EN.md`：公开说明与快速开始
- `DESIGN.md`：工作流设计与取舍说明
- `INSTALL.md`：本地安装与 GitHub 发布说明
- `prompts/`：可复用提示词片段
- `tools/`：本地辅助脚本
- `examples/`：真实场景示例调用链

若缺少这些内容，Skill 仍可运行，但不适合作为公开仓库分发。

若用户只提供模板文件但未提供清晰文字规则：
- 可以辅助阅读模板
- 可以抽取可见结构特征
- 但不得声称已经完整掌握模板规范
- 不得直接开始正式排版或终稿生成

## 运行模式

### `--assist`
- 只分析与建议
- 不生成正式稿
- 不覆盖关键文件

### `--draft`
- 生成草稿
- 不默认覆盖已定稿内容
- 适合提纲、章节初稿、核查报告

### `--auto`
- 自主审阅项目
- 自主收集证据与图表
- 自主更新 notes / refs / 草稿文件
- 仅在 requirements 已明确后可进入正式写作链

### `--team`
- 启动多 Agent 协同
- 适合多章并行或长链任务

### `--team-strict`
- 强制拆任务、分文件所有权、集中汇总
- 用于整篇论文或多章重写

### `--safe`
- 用于审查/降重等保守操作
- 以保留事实锚点为先

### `--aggressive`
- 用于高强度改写/降重
- 但仍不得修改实验结论与核心指标

## 能力声明（MCP Types）

本 Skill 可以调用或映射以下类型的 MCP 能力：
- **需求采集类**：向用户追问结构、格式、引用、交付要求
- **搜索类**：检索论文、会议、期刊、GitHub、标准规范
- **网页抓取类**：抓取文献元数据、网页正文、会议页、DOI 信息
- **代码审阅类**：分析项目结构、训练脚本、推理逻辑、Web 系统实现
- **本地执行类**：运行训练、验证、推理、统计、文档生成脚本
- **文件系统类**：扫描结果目录、读取日志、收集图表与输出文件
- **浏览器/截图类**：打开本地 Web 系统、自动操作、截图论文素材
- **图像处理类**：裁剪、拼接、统一尺寸、生成图版
- **文档解析类**：解析 Markdown、PDF、docx、模板说明文档
- **表格/数据类**：从 csv/json/log 汇总实验表与对比表
- **Office/文档校验类**：检查目录、页码、标题样式、导出 PDF
- **文本审计类**：检查术语漂移、幻觉、高风险措辞、重复表述

## 强制前置闸门：`/thesis-intake`

### 目标
开始写论文前，详细向用户收集全部关键要求。

### 硬约束
未完成 intake 前：
- `/thesis-write` 只能运行在 `--assist` 或 `--draft`
- `/thesis-format` 不得进入正式排版
- `/thesis-build` 不得生成正式终稿

### 必采集信息（8 组）

#### 1. 基本信息
- 论文题目
- 学校 / 学院 / 专业
- 学历层次（本科 / 硕士 / 其他）
- 作者姓名
- 指导教师
- 截止时间
- **总字数要求**（学校规定的正文字数范围）
- **论文类型倾向**（算法为主 / 系统为主 / 均衡型，若用户不确定则由 Skill 根据项目内容判断）

#### 2. 论文结构要求
- 是否固定五章/六章
- 以下为**默认必要项**（除非用户明确排除）：中文摘要、Abstract、目录、结论、致谢、参考文献
- 是否需要：前言、附录
- 各章是否有固定命名

#### 3. 写作范围与来源
- 是基于现有项目反推论文，还是从零研究型写作
- 可用材料：代码、日志、图表、实验报告、开题报告、任务书、PPT、系统截图等
- 哪些内容已定稿，哪些需要重写

#### 4. 排版格式要求
- 页面大小
- 页边距
- 正文字体字号
- 标题字体字号
- 行距、段前段后
- 首行缩进
- 页码位置与编号方式
- 是否分节

#### 5. 标题编号规则
- 一级标题格式
- 二级标题格式
- 三级标题格式
- 是否允许四级标题
- 目录显示到几级

#### 6. 图表公式规则
- 图题在上还是在下
- 表题在上还是在下
- 图号 / 表号格式
- 公式是否编号
- 三线表要求
- 图片宽度/居中/注释要求

#### 7. 参考文献规则
- 使用哪套标准（如 GB/T 7714-2015、APA、IEEE 等）
- 编号制还是作者年制
- **文内引用标记格式**（如 `[1]`、`[1,2]`、`[1-3]`、`(张三, 2024)`、上标 `[1]` 等）
- 文内引用标记位置（句末标点前还是标点后、上标还是平标）
- 文后排序规则
- 是否允许 GitHub / arXiv / 网页引用
- 是否要求访问日期
- 参考文献条目必需字段（作者、题名、期刊/会议、年份、卷号、页码、DOI 等）

#### 8. 终稿交付要求
- 需要 docx / pdf / 查重稿 / 盲审稿 / 答辩稿 哪些版本
- 是否需要封面、承诺书、任务书合并
- 是否需要答辩 PPT

### 模板要求持久化（硬约束）

用户提供的所有格式要求、模板说明、学校规范，必须持久化写入 `thesis/notes/template_requirements.md`。

- 每次会话启动、每个子 Skill 执行前，必须先读取此文件
- 不得仅依赖上下文记忆模板要求——长任务中上下文会丢失
- 用户补充或修改格式要求时，必须同步更新此文件
- 此文件是格式相关决策的**唯一真相源**，任何格式决策与此文件冲突时以此文件为准

### 输出物
- `thesis/notes/intake_requirements.md`
- `thesis/notes/missing_requirements.md`
- `thesis/notes/template_requirements.md`（模板格式要求持久化文件）

### 缺失清单模板

```md
## 论文写作前仍缺以下信息

### 必须补充
- 标题编号规则
- 正文字体字号与行距
- 图题/表题位置规则
- 参考文献格式标准
- 是否需要中英文摘要与致谢

### 建议补充
- 各章目标字数
- 目录显示层级
- 页码规则
- 公式编号规则

### 说明
我无法仅凭模板文件完整、准确提取全部排版细则。
请提供文字版格式要求，或将模板要求整理成可执行条目。
```

## `/thesis-data` — 撰写前期数据准备

### 目标
- 自主审阅项目
- 运行项目或统计脚本获取论文支撑数据
- 收集模型结果、日志、环境信息、截图素材
- **收集引用论文与文献资源（强制）**

### 论文引用收集（硬约束）

本阶段必须完成引用论文的收集与准备。**引用论文未准备就绪前，禁止进入 `/thesis-write`。**

#### 收集策略
1. 根据论文主题与研究方向，确定需要引用的核心文献领域
2. 优先搜索国外论文（arXiv、IEEE、ACM、Springer、Elsevier 等），国内论文因下载渠道限制优先级较低
3. 每篇目标文献必须尝试以下步骤：
   a. 搜索论文元数据（标题、作者、年份、会议/期刊、DOI）
   b. 尝试下载原文 PDF（调用一切可用工具：WebFetch、WebSearch、mcp 工具等）
   c. 下载成功 → 存入 `thesis/refs/papers/`，记录元数据
   d. 下载失败 → 标记为 `need-manual-download`，通知用户手动下载
4. 收集每篇论文的关键数据、图表、结论，用于支撑正文论证

#### 文献状态
- `downloaded`：原文已下载到本地
- `metadata-only`：仅有元数据，原文待下载
- `need-manual-download`：无法自动下载，需用户手动获取
- `placeholder`：用户指定但尚未检索的文献

#### 通知用户的时机
- 当有文献标记为 `need-manual-download` 时，立即通知用户，列出：
  - 论文标题、作者、年份
  - 建议的下载来源（如 DOI 链接、arXiv ID）
  - 期望用户完成下载的时间节点

### 允许调用的 MCP 能力
- 代码审阅类
- 本地执行类
- 文件系统类
- 浏览器/截图类
- 表格/数据类
- 图像处理类
- 系统环境类
- **搜索类**（论文检索）
- **网页抓取类**（论文元数据与原文下载）

### 自动动作建议
- 扫描训练脚本、推理脚本、配置、实验报告、结果目录
- 生成指标汇总表
- 抽取环境信息（Python、CUDA、GPU、依赖版本）
- 启动本地 Web demo 并抓取论文可用截图
- **根据论文主题检索相关文献，建立引用清单**
- **尝试下载文献原文到本地**
- **提取已下载文献的关键数据与图表**

### 输出物
- `thesis/notes/data_inventory.md`
- `thesis/notes/experiment_summary.md`
- `thesis/notes/runtime_env.md`
- `thesis/notes/metric_tables.md`
- `thesis/figures/`
- `thesis/refs/papers_inventory.md`（文献清单与状态）
- `thesis/refs/papers_to_download.md`（需用户手动下载的文献列表）
- `thesis/refs/papers/`（已下载的文献原文）
- `thesis/refs/paper_data_extracts.md`（从文献中提取的关键数据）

## `/thesis-outline` — 正式撰写前框架搭建

### 目标
- 先定章节边界
- 给每章绑定数据源、图表源、代码源、文献源
- **根据论文类型制定字数预算**
- 锁定写作范围

### 字数预算（硬约束）

不同类型的论文，各章字数分配差异很大。制定大纲时必须根据论文类型确定字数预算，写入 `thesis_outline.md`。

#### 论文类型识别

| 类型 | 判定条件 | 特征 |
|------|----------|------|
| **算法为主型** | 核心贡献是模型/算法设计，实验占比大 | 第三章偏短（系统轻量），第四章是重点 |
| **系统为主型** | 核心贡献是系统设计与工程实现 | 第三章是重点（架构+模块+交互），第四章偏短 |
| **算法+系统均衡型** | 既有核心算法改进，又有完整系统实现 | 三四章字数接近，整体偏长 |

#### 各学历层次总字数参考

| 学历 | 总字数（正文） | 章数 |
|------|---------------|------|
| 专科 | 8000-12000 | 4-5章 |
| 本科 | 15000-25000 | 5章 |
| 硕士（专硕） | 25000-35000 | 5-6章 |
| 硕士（学硕） | 30000-50000 | 5-6章 |

> 注：以上为国内常见范围，**以用户学校要求为准**。用户未明确时，按上表默认值规划。

#### 各章字数分配参考

以下为**本科 5 章论文**的分配参考，总字数约 20000 字：

| 章 | 内容 | 算法为主型 | 系统为主型 | 均衡型 |
|----|------|-----------|-----------|--------|
| 一 | 绪论 | 2500-3500 | 2500-3500 | 2500-3500 |
| 二 | 相关技术与理论基础 | 3000-4000 | 2500-3500 | 3000-4000 |
| 三 | 系统设计与实现 | 2500-3500 | **4000-5500** | 3500-4500 |
| 四 | 实验与结果分析 | **5000-6500** | 2500-3500 | 4000-5000 |
| 五 | 总结与展望 | 1000-1500 | 1000-1500 | 1000-1500 |

**核心规则**：
- 论文重心在哪章，那章字数占比应达到 **25%-35%**
- 摘要 300-500 字，Abstract 200-400 词
- 致谢 300-500 字
- 绪论中"研究现状"占比最大（约占本章 50%-60%）
- 实验章中"结果分析"占比最大（约占本章 60%-70%）

#### 字数规划输出格式

大纲中必须包含如下格式的字数规划表：

```markdown
## 字数规划

论文类型：[算法为主型/系统为主型/均衡型]
目标总字数：XXXXX 字

| 章 | 目标字数 | 占比 | 写作重点 |
|----|---------|------|---------|
| 第一章 绪论 | XXXX | XX% | ... |
| 第二章 理论基础 | XXXX | XX% | ... |
| 第三章 系统设计 | XXXX | XX% | ... |
| 第四章 实验分析 | XXXX | XX% | ... |
| 第五章 总结展望 | XXXX | XX% | ... |
| 摘要+致谢 | XXXX | — | ... |

非正文部分：参考文献 XX 篇、图 XX 张、表 XX 个
```

### 允许调用的 MCP 能力
- 代码审阅类
- 搜索类
- 文档解析类
- 表格类

### 输出物
- `thesis/notes/thesis_outline.md`
- `thesis/notes/chapter_matrix.md`
- `thesis/notes/figure_plan.md`
- `thesis/notes/reference_plan.md`

### 写作边界铁律
- 第一章：为什么做
- 第二章：理论依据
- 第三章：怎么做
- 第四章：效果如何
- 第五章：做成了什么、还差什么

## `/thesis-assets` — 图表/截图/表格资产管理

### 目标
- 独立管理图、表、截图、案例图、系统界面图
- 防止图号错位、错图、漏图、低质量展示图进入终稿

### 允许调用的 MCP 能力
- 浏览器/截图类
- 图像处理类
- 文件系统类
- 表格生成类
- OCR/图像检查类

### 图表生成策略（硬约束）

本 Skill 将图表分为两类，生成方式截然不同，不得混淆：

#### A. 概念图 / 逻辑图（提示词生成）
- 类型：流程图、架构图、系统设计图、模块关系图、数据流图、部署图等
- 生成方式：生成**图片描述提示词**，供外部 AI 绘图工具（如 Mermaid、draw.io、AI 图像生成）或手动绘制使用
- 输出：`thesis/figures/prompts/<figure_name>_prompt.md`
- 不得声称已生成实际图片，必须明确标注为”提示词”

#### B. 实验数据图（代码生成）
- 类型：实验结果图、指标对比图、损失曲线、混淆矩阵、ROC 曲线、消融实验图、性能柱状图等
- 生成方式：使用 **Python 脚本**（matplotlib、seaborn、plotly 等）从实际实验数据生成
- 数据来源：必须是 `thesis-data` 阶段收集的真实实验数据，不得编造
- 输出：`thesis/figures/<figure_name>.png` + 对应的 `thesis/figures/scripts/<figure_name>.py`
- 脚本必须可重复执行，便于数据更新后重新生成

#### C. 截图类（实际捕获）
- 类型：系统界面截图、运行结果截图、Web 演示截图等
- 生成方式：浏览器截图工具或系统截图
- 输出：`thesis/figures/<figure_name>.png`

### 硬规则
1. 每章单独维护图号范围
2. 图号首次在正文出现时插图
3. 同章同图只插一次
4. 图号变更必须同步修改正文、映射、图题
5. 覆盖旧图前先备份
6. 图像必须经过”答辩可读性”人工验收
7. **概念图必须先生成提示词，实验数据图必须用 Python 脚本从真实数据生成，二者不可混用**

### 输出物
- `thesis/figures/*.png`
- `thesis/notes/assets_manifest.md`

## `/thesis-write` — 论文正文撰写

### 前置条件
- 必须已完成 `/thesis-intake`
- 必须已读取 `thesis/notes/template_requirements.md`（模板格式要求）
- 若关键格式要求缺失，只能 `--assist` / `--draft`
- **资源就绪闸门（硬约束）**：以下资源必须准备完毕，否则禁止进入正文撰写：
  - 引用论文已收集且至少 `metadata-only` 状态
  - 实验数据已汇总到 `metric_tables.md`
  - 图表计划已明确到 `figure_plan.md`
  - 各章证据源已绑定到 `chapter_matrix.md`

### 目标
- 按章生成 Markdown 正文
- 证据驱动写作
- 避免长上下文污染与章节互串
- **正文与图表强耦合，边写边插图**

### 正文与图表耦合规则（硬约束）

正文撰写时必须与前期准备的资源紧密结合，**禁止先写正文后插图**：

1. **写前规划**：每章动笔前，先确认本章涉及的图号、表号、引用编号、证据源
2. **边写边插图**：写到需要图表的位置时，立即插入图号标记与图题，确保图与正文内容对应
3. **缺图即补**：写作中发现缺少必要的图表，立即触发补图流程：
   - 概念图 → 生成提示词到 `thesis/figures/prompts/`
   - 实验数据图 → 编写 Python 脚本生成到 `thesis/figures/`
   - 截图 → 调用截图工具捕获
4. **引用同步**：正文引用文献时，同步检查 `papers_inventory.md` 中该文献的状态，未下载的文献标注提醒
5. **图号连贯**：新增图表时必须更新 `figure_plan.md` 与 `assets_manifest.md`，保持图号连贯

### 降低 AI 率写作规范（硬约束）

AI 生成论文的典型特征是过度结构化、连接词机械重复、段落对称均匀，导致 AI 检测率和查重率双高。正文撰写时必须遵守以下规则：

#### 禁止的写作模式

以下模式在正文中**单章出现不得超过 2 次**，连续两段不得使用相同模式：

1. **机械递进**：`"首先……其次……再次……此外……最后……"` 连续铺排超过 3 项
2. **对称编号**：`"第一……第二……第三……第四……"` 大段并列（总结章的要点概述除外）
3. **其一其二**：`"其一……其二……其三……"` 用于段落内部论证
4. **空洞过渡**：`"随着XX的发展，XX越来越受到关注"` `"值得注意的是"` `"需要强调的是"` `"不可否认"` `"综上所述"` 等 AI 高频套话
5. **段段总分总**：每段都是"论点句→展开→总结句"的完美三段式
6. **等长段落**：连续 3 段以上字数接近（视觉上形成"豆腐块"）

#### 必须遵守的写作要求

1. **段落长短交替**：长段（200-400字）与短段（50-100字）穿插，打破视觉均匀感
2. **自然过渡**：用内容本身的逻辑关系衔接，而非依赖连接词。上下段落之间应有因果关系、递进关系或转折关系，但不一定要显式写出"因此"、"然而"
3. **具体事实替代抽象总结**：用具体数据、实验结果、代码引用代替"效果显著"、"性能提升明显"等空话
4. **有详有略**：核心贡献详写（方法细节+实验支撑），辅助内容略写（一段带过），不要平均用力
5. **术语替代通用表述**：用项目特有的术语（如"UNet++ B4"、"Focal Tversky Loss"、"DH 数据集"）替代通用词汇（如"该模型"、"本文方法"、"实验数据集"）
6. **引用自然嵌入**：在叙述具体方法时紧跟引用编号，不单独列引用句。格式：`"方法描述[编号]"`
7. **图文交叉**：纯文字连续不超过 3 段，中间穿插图表引用打破文本墙
8. **偶尔断句或短句**：在长论述中插入一句简短判断或评价，增加节奏变化
9. **不回避缺陷**：实验分析中主动指出不足，比一味正面论述更可信
10. **本章小结简洁**：每章结尾的"本章小结"控制在 3-5 句，只写本章做了什么、为下章铺垫什么，不复述细节

#### 自检清单

每章写完后，执行以下检查：
- [ ] 是否存在连续 3 段以上使用相同结构化连接词？
- [ ] 是否有 3 段以上字数近似（偏差 < 20%）？
- [ ] 是否有"值得注意的是"、"需要强调的是"等 AI 套话？
- [ ] 是否有空泛表述（"效果显著"）无数据支撑？
- [ ] 纯文字是否超过 3 段未插入图表或引用？
- [ ] 引用是否自然嵌入叙述中而非单独成句？

#### 写作节奏参考

```text
好：长段（机制分析）→ 短段（一句评价）→ 图表引用+解读 → 中段（实验支撑）→ 短段（过渡）
差：中段 → 中段 → 中段 → 中段 → 每段都是总分总
```

### 推荐写作顺序
1. 第五章
2. 第二章
3. 第三章
4. 第四章
5. 第一章
6. 摘要 / Abstract

## 历史问题尸账

### 1. 标题编号重复
- 现象：Markdown 已写 `1.1`，脚本又自动补编号
- 根因：未先剥离已有编号
- 规避：进入生成层前统一 `strip_heading_number`

### 2. 标题体系与模板不一致
- 现象：生成的是 `第一章/1.1`，模板要求 `一、/（一）/1.`
- 根因：按技术文档习惯实现，不是按用户模板实现
- 规避：一切以 intake 明确后的文字规则为准

### 3. 插图触发条件过窄导致漏图
- 规避：图号首次出现即插图，不依赖固定措辞

### 4. 图号顺延后正文与映射脱节
- 规避：正文引用、映射、图题三处一起改

### 5. 新图已放入目录，但论文仍在用旧图
- 规避：优先覆盖旧文件名，否则同步修改映射

### 6. 图像展示性差
- 规避：必须做答辩可读性验收

### 7. Word 占用导致保存失败
- 规避：正式生成前关闭 docx，必要时输出 `_验收版.docx`

### 8. 参考文献格式与模板冲突
- 规避：以用户明确给出的文字规则为准，不擅自默认

### 9. 前置页与正文格式混杂
- 规避：封面、摘要、目录、正文、参考文献、致谢分段处理

### 10. 公式、表格、图题规则未提前锁定
- 规避：开工当天在 intake 中明确

### 11. 长上下文导致写作目标漂移
- 规避：多章写作优先 Team mode，拆章并行，主控统一收口

### 12. AI 编造文献或实验结论
- 规避：通过 `/thesis-citations` 与 `/thesis-audit` 强制联网核验

### 13. 前期数据准备缺少论文引用收集
- 现象：进入正文撰写时没有可用引用论文，导致引用空洞
- 根因：`/thesis-data` 未包含文献收集环节
- 规避：`/thesis-data` 强制要求论文引用收集，未就绪禁止进入 `/thesis-write`；优先国外论文便于下载；下载失败立即通知用户

### 14. 正文撰写与前期资源解耦
- 现象：正文写完后再插图，导致图与内容不对应、缺图、图号混乱
- 根因：写作与图表准备是独立步骤，未在写作时内联插图
- 规避：正文撰写时边写边插图，写前确认本章图号/表号/引用，缺图即补；资源未齐备不得动笔

### 15. 模板要求仅存于上下文，长任务中遗忘
- 现象：中后期写作偏离论文模板要求，格式错误频出
- 根因：格式要求未持久化，仅靠上下文记忆
- 规避：intake 阶段将所有格式要求写入 `template_requirements.md`，每次会话/子 Skill 启动时必须先读取此文件

### 16. 概念图与实验数据图生成策略混淆
- 现象：要求 Python 生成架构图（不合理），或用 AI 提示词生成实验曲线（不可复现）
- 根因：未区分两类图表的生成方式
- 规避：概念图（流程图/架构图）生成提示词；实验数据图用 Python 脚本从真实数据生成；二者不可混用

### 17. AI 率过高——过度结构化写作
- 现象：AI 检测率和查重率双高，全文充满"首先/其次/再次/此外/最后"机械递进、段落等长对称、每段总分总
- 根因：AI 默认倾向生成结构化、对称、均匀的文本，这是 AI 写作最显著的指纹
- 规避：段落长短交替、自然过渡不依赖连接词、有详有略、具体数据替代空话、术语替代通用表述、图文交叉打破文本墙

### Agent Team 提示词铁律

#### `lead`
- 只做总控、分章、统一术语与口径
- 不直接并发改同一章节文件
- 汇总时必须输出：章节状态、证据覆盖、剩余风险

#### `researcher`
- 只查证据，不扩写结论
- 必须回报：代码/日志/图表/文献来源路径
- 对未证实内容必须标 `unverified`

#### `writer`
- 只改分配章节文件
- 不得跨章改标题、图号、引用编号
- 每段关键结论后要能追溯到证据源

#### `citation-checker`
- 只核引用真实性与编号闭环
- 不负责润色正文
- 必须标注文献状态：`verified/partial/unverified/fake-risk`

#### `reviewer`
- findings 优先
- 无问题明确写 `no findings`
- 只审错，不扩写，不重构

### Team 启动模板

```text
角色：<lead/researcher/writer/citation-checker/reviewer>
目标：<本轮唯一目标>
可写文件：<仅列分配文件>
禁止事项：不得扩域；不得修改未分配文件；不得编造证据
交付必须包含：
1. 完成内容
2. 使用的证据源
3. 未证实内容
4. 验证命令或核验动作
```

### Team 收敛清单
- 是否存在多 agent 同写一文件
- 是否有未回报证据源的段落
- 是否有 `fake-risk` 文献未处置
- 是否有图号/表号改动未同步正文
- 是否有结论超出实验支撑

## 最终目标

这个 Skill 的目标不是“快速拼一篇论文”，而是：
- 在要求明确前不乱写
- 在证据不明前不乱断言
- 在文献未核实前不乱引用
- 在排版规则不清前不乱生成
- 在长任务中用 Team 降低上下文污染与目标漂移
- 在查重优化时保住事实锚点与实验结论
