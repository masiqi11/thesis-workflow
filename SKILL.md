# Thesis Workflow

## 仓库定位

这是一个面向 Claude Code 的 GitHub-ready Skill 仓库。

其目标不是把论文写作变成“黑盒一键生成”，而是把高风险长链任务拆成：
- intake 闸门
- 证据收集
- 提纲与资产规划
- 章节写作
- 内容核对
- 引用真值核验
- 格式审查
- AI 风险审查
- 降重优化
- 终稿构建

若将本仓库公开发布，应把它理解为：**workflow + prompt assets + helper tools + examples**。

## 总则

这是**通用型论文写作 Skill**，不是针对某一学校模板硬编码的固定脚本。

### 第一铁律
**未澄清要求，不得开始正式论文编写。**

在以下信息未明确前，本 Skill 不得直接进入正式正文撰写：
- 学校/学院/专业要求
- 论文结构要求
- 标题编号规则
- 字体字号
- 行距段距
- 页边距
- 页码规则
- 图题表题规则
- 公式规则
- 参考文献格式
- 摘要/Abstract/致谢/附录要求

### 模板限制说明
我无法仅凭一个 `doc/docx` 模板文件，稳定、完整、准确地提取全部学校排版细则。
若模板中的格式要求对最终稿有约束，必须由用户提供**文字版格式要求**，或将模板要求整理成可执行条目。

### 仓库配套资产
本仓库建议同时携带以下配套内容：
- `README.md` / `README_EN.md`：公开说明与快速开始
- `DESIGN.md`：工作流设计与取舍说明
- `INSTALL.md`：本地安装与 GitHub 发布说明
- `prompts/`：可复用提示词片段
- `tools/`：本地辅助脚本
- `examples/`：真实场景示例调用链

若缺少这些内容，Skill 仍可运行，但不适合作为公开仓库分发。

若用户只提供模板文件但未提供清晰文字规则：
- 可以辅助阅读模板
- 可以抽取可见结构特征
- 但不得声称已经完整掌握模板规范
- 不得直接开始正式排版或终稿生成

## 运行模式

### `--assist`
- 只分析与建议
- 不生成正式稿
- 不覆盖关键文件

### `--draft`
- 生成草稿
- 不默认覆盖已定稿内容
- 适合提纲、章节初稿、核查报告

### `--auto`
- 自主审阅项目
- 自主收集证据与图表
- 自主更新 notes / refs / 草稿文件
- 仅在 requirements 已明确后可进入正式写作链

### `--team`
- 启动多 Agent 协同
- 适合多章并行或长链任务

### `--team-strict`
- 强制拆任务、分文件所有权、集中汇总
- 用于整篇论文或多章重写

### `--safe`
- 用于审查/降重等保守操作
- 以保留事实锚点为先

### `--aggressive`
- 用于高强度改写/降重
- 但仍不得修改实验结论与核心指标

## 能力声明（MCP Types）

本 Skill 可以调用或映射以下类型的 MCP 能力：
- **需求采集类**：向用户追问结构、格式、引用、交付要求
- **搜索类**：检索论文、会议、期刊、GitHub、标准规范
- **网页抓取类**：抓取文献元数据、网页正文、会议页、DOI 信息
- **代码审阅类**：分析项目结构、训练脚本、推理逻辑、Web 系统实现
- **本地执行类**：运行训练、验证、推理、统计、文档生成脚本
- **文件系统类**：扫描结果目录、读取日志、收集图表与输出文件
- **浏览器/截图类**：打开本地 Web 系统、自动操作、截图论文素材
- **图像处理类**：裁剪、拼接、统一尺寸、生成图版
- **文档解析类**：解析 Markdown、PDF、docx、模板说明文档
- **表格/数据类**：从 csv/json/log 汇总实验表与对比表
- **Office/文档校验类**：检查目录、页码、标题样式、导出 PDF
- **文本审计类**：检查术语漂移、幻觉、高风险措辞、重复表述

## 强制前置闸门：`/thesis-intake`

### 目标
开始写论文前，详细向用户收集全部关键要求。

### 硬约束
未完成 intake 前：
- `/thesis-write` 只能运行在 `--assist` 或 `--draft`
- `/thesis-format` 不得进入正式排版
- `/thesis-build` 不得生成正式终稿

### 必采集信息（8 组）

#### 1. 基本信息
- 论文题目
- 学校 / 学院 / 专业
- 学历层次（本科 / 硕士 / 其他）
- 作者姓名
- 指导教师
- 截止时间

#### 2. 论文结构要求
- 是否固定五章/六章
- 是否需要：前言、摘要、Abstract、目录、结论、致谢、参考文献、附录
- 各章是否有固定命名

#### 3. 写作范围与来源
- 是基于现有项目反推论文，还是从零研究型写作
- 可用材料：代码、日志、图表、实验报告、开题报告、任务书、PPT、系统截图等
- 哪些内容已定稿，哪些需要重写

#### 4. 排版格式要求
- 页面大小
- 页边距
- 正文字体字号
- 标题字体字号
- 行距、段前段后
- 首行缩进
- 页码位置与编号方式
- 是否分节

#### 5. 标题编号规则
- 一级标题格式
- 二级标题格式
- 三级标题格式
- 是否允许四级标题
- 目录显示到几级

#### 6. 图表公式规则
- 图题在上还是在下
- 表题在上还是在下
- 图号 / 表号格式
- 公式是否编号
- 三线表要求
- 图片宽度/居中/注释要求

#### 7. 参考文献规则
- 使用哪套标准
- 编号制还是作者年制
- 正文引用格式
- 文后排序规则
- 是否允许 GitHub / arXiv / 网页引用
- 是否要求访问日期

#### 8. 终稿交付要求
- 需要 docx / pdf / 查重稿 / 盲审稿 / 答辩稿 哪些版本
- 是否需要封面、承诺书、任务书合并
- 是否需要答辩 PPT

### 输出物
- `thesis/notes/intake_requirements.md`
- `thesis/notes/missing_requirements.md`

### 缺失清单模板

```md
## 论文写作前仍缺以下信息

### 必须补充
- 标题编号规则
- 正文字体字号与行距
- 图题/表题位置规则
- 参考文献格式标准
- 是否需要中英文摘要与致谢

### 建议补充
- 各章目标字数
- 目录显示层级
- 页码规则
- 公式编号规则

### 说明
我无法仅凭模板文件完整、准确提取全部排版细则。
请提供文字版格式要求，或将模板要求整理成可执行条目。
```

### 目标

## `/thesis-write` — 论文正文撰写

### 前置条件
- 必须已完成 `/thesis-intake`
- 若关键格式要求缺失，只能 `--assist` / `--draft`

### 目标
- 按章生成 Markdown 正文
- 证据驱动写作
- 避免长上下文污染与章节互串

### 推荐顺序
1. 第五章
2. 第二章
3. 第三章
4. 第四章
5. 第一章
6. 摘要 / Abstract

### 多 Agent Team 机制（强建议）
满足任一条件时，应优先创建 Agent Team：
- 一次写 2 章及以上
- 单章需要同时查代码、实验、文献、截图
- 用户要求从零完整写论文
- 当前上下文过长，存在污染风险
- 需要并行产出与并行复核

### Team 角色建议
- `lead`：总控章节边界与统一口径
- `researcher`：查代码 / 实验 / 日志 / 配置
- `writer`：写指定章节
- `citation-checker`：补引用位与文献占位
- `reviewer`：只审错，不扩写

### 文件所有权铁律
同一时刻只允许一个 agent 改一个章节文件。
不得多个 agent 同时写同一章。

### 每个 agent 交付时必须回报
- 本章目标
- 使用的证据源
- 未证实内容列表
- 需主控复核的问题点

### 允许调用的 MCP 能力
- 代码审阅类
- 实验结果读取类
- 搜索类
- 文档解析类
- Agent/Team orchestration 类

### 输出物
- `thesis/*.md`
- `thesis/notes/chapter_evidence_map.md`

核对论文内部逻辑、事实、术语、参数、指标、章节口径。

### 允许调用的 MCP 能力
- 代码审阅类
- 本地执行类

- 差异分析类

### 核查重点
- 模型名称是否一致

- 参数设置是否一致
- 指标数值是否一致
- 系统功能是否与真实代码匹配

### 输出物

## `/thesis-citations` — 论文引用核对

### 目标
不仅核对编号格式，还要核验文献是否**真实存在**。

### 必须调用的 MCP 能力
- 搜索类
- 网页抓取类
- 文献元数据解析类
- 正文扫描类

### 每条文献必须做的动作
1. 搜索标题
2. 搜索作者 + 年份
3. 校验会议/期刊名称
4. 校验 DOI / arXiv / GitHub / URL
5. 判断是否真实可检索

### 文献状态
- `verified`
- `partial`
- `unverified`
- `fake-risk`

### 核查重点
- 正文 `[n]` 是否都有对应条目
- 文后条目是否至少被引用一次
- 题目、作者、年份、会议是否匹配
- 在线资源是否含访问日期
- 可疑文献是否被标为 `fake-risk`

### 输出物
- `thesis/notes/citation_audit.md`
- `thesis/notes/reference_truth_report.md`
- `thesis/refs/references_checked.md`
- 可选：`thesis/refs/references.bib`

## `/thesis-audit` — AI 风险与全文总审查

### 目标
专项审查 AI 写论文高风险问题。

### 必须调用的 MCP 能力
- 搜索 MCP
- Web 抓取 MCP
- 代码审阅 MCP
- 执行 MCP
- 文本审计 MCP

### 必查问题
#### A. 幻觉类
- 论文声称实现了某功能，但代码中没有
- 论文声称做了某实验，但结果目录无证据
- 论文声称使用某模块，但项目内不存在

#### B. 虚空引用类
- 文中引用的论文检索不到
- 标题/作者/年份/会议不匹配
- arXiv / GitHub / DOI 不存在
- 文中有 `[n]`，文后无对应条目
- 文后有条目，正文从未引用

#### C. 数值类
- 摘要、正文、结论中的指标不一致
- 图中数值与表格数值不一致
- 最优模型描述前后矛盾

#### D. 术语类
- 模型名混写
- 指标名混写
- 数据集名混写
- 中文术语漂移

#### E. 逻辑类
- 理论与实现脱节
- 系统设计与实验验证不对应
- 总结夸大超出实验支撑

### 风险分级
- `P0`：虚假引用、虚假实验、虚假实现
- `P1`：关键指标不一致、图表解释错误
- `P2`：术语不统一、措辞空泛

### 输出物
- `thesis/notes/ai_risk_audit.md`
- `thesis/notes/claim_evidence_matrix.md`

## `/thesis-reduce` — 降重优化

### 目标
针对查重做定向改写，但保留事实、结论、指标、引用与术语锚点。

### 核心原则
- 不改实验事实
- 不改指标数值
- 不改核心模型名/数据集名/参数名
- 不删除关键引用
- 不用空话替换技术内容

### 适合优化的对象
- 定义段
- 背景综述段
- 模型介绍通用段
- 与实验报告/PPT重复的系统描述
- 空泛套话

### 常用降重策略
1. 句法改写
2. 结构改写
3. 保留术语、重写周边叙述
4. 用项目事实替代模板化空话
5. 压缩冗长重复表述

### 禁止动作
- 为降重篡改结论
- 删除关键实验说明
- 把引用删没
- 把专业术语改错

### 模式
- `--safe`
- `--target chapter=<n>`
- `--aggressive`

### 输出物
- `thesis/notes/reduce_report.md`
- 标注“原句 / 改写 / 保留事实点”

## `/thesis-sync` — 结果同步与归档

### 目标
统一同步论文、图表、文献、终稿、答辩材料与 notes。

### 动作
- 同步实验结果到正文或 notes
- 同步新图到 `FIG_MAP` 或资产清单
- 同步文献到引用清单
- 同步终稿到 `release/`

---

## 证据矩阵（横切硬约束）

每个关键结论都要挂证据，格式建议：

```text
结论/说法 -> 证据来源 -> 验证状态 -> 所在章节
```

示例：
- `UNet++ B4 为最佳单模型` -> 实验结果表 / 图 -> `verified` -> 第四章
- `系统支持阈值调节` -> 代码实现 / 截图 -> `verified` -> 第三章

## 历史问题尸账

### 1. 标题编号重复
- 现象：Markdown 已写 `1.1`，脚本又自动补编号
- 根因：未先剥离已有编号
- 规避：进入生成层前统一 `strip_heading_number`

### 2. 标题体系与模板不一致
- 现象：生成的是 `第一章/1.1`，模板要求 `一、/（一）/1.`
- 根因：按技术文档习惯实现，不是按用户模板实现
- 规避：一切以 intake 明确后的文字规则为准

### 3. 插图触发条件过窄导致漏图
- 规避：图号首次出现即插图，不依赖固定措辞

### 4. 图号顺延后正文与映射脱节
- 规避：正文引用、映射、图题三处一起改

### 5. 新图已放入目录，但论文仍在用旧图
- 规避：优先覆盖旧文件名，否则同步修改映射

### 6. 图像展示性差
- 规避：必须做答辩可读性验收

### 7. Word 占用导致保存失败
- 规避：正式生成前关闭 docx，必要时输出 `_验收版.docx`

### 8. 参考文献格式与模板冲突
- 规避：以用户明确给出的文字规则为准，不擅自默认

### 9. 前置页与正文格式混杂
- 规避：封面、摘要、目录、正文、参考文献、致谢分段处理

### 10. 公式、表格、图题规则未提前锁定
- 规避：开工当天在 intake 中明确

### 11. 长上下文导致写作目标漂移
- 规避：多章写作优先 Team mode，拆章并行，主控统一收口

### 12. AI 编造文献或实验结论
- 规避：通过 `/thesis-citations` 与 `/thesis-audit` 强制联网核验

### 子 Skill 输入/输出契约

#### `/thesis-intake`
- 输入：用户口述要求、学校规范、模板说明、文字版格式条目
- 输出：`intake_requirements.md`、`missing_requirements.md`
- 通过条件：关键格式规则已明确，缺失项不含“必须补充”

#### `/thesis-data`
- 输入：项目代码、实验日志、结果目录、截图入口、环境信息
- 输出：`data_inventory.md`、`experiment_summary.md`、`metric_tables.md`、`runtime_env.md`
- 通过条件：关键实验指标、数据集规模、运行环境、可用图表已落盘

#### `/thesis-outline`
- 输入：intake 结果、项目证据、材料清单
- 输出：`thesis_outline.md`、`chapter_matrix.md`、`figure_plan.md`、`reference_plan.md`
- 通过条件：每章目标、证据源、图表源、参考文献源已绑定

#### `/thesis-assets`
- 输入：图表原图、截图、表格数据、图号规划
- 输出：`assets_manifest.md`、规范化后的 `thesis/figures/*`
- 通过条件：图号连续、图题明确、正文引用可追踪、低质量图片已剔除

#### `/thesis-write`
- 输入：outline、证据矩阵、图表计划、章节目标字数
- 输出：章节 Markdown、`chapter_evidence_map.md`
- 通过条件：每章均附证据来源，未证实内容已显式标记

#### `/thesis-content`
- 输入：全文章节、代码实现、实验结果、系统截图
- 输出：`content_audit.md`
- 通过条件：模型名、数据集、参数、指标、系统功能口径一致

#### `/thesis-citations`
- 输入：正文引用、参考文献列表、联网搜索结果
- 输出：`citation_audit.md`、`reference_truth_report.md`、`references_checked.md`
- 通过条件：不存在 `fake-risk` 未处置条目，正文与文后引用双向闭合

#### `/thesis-format`
- 输入：Markdown 章节、图表资产、明确的文字版格式规则、模板文件（可选）
- 输出：Word 验收版、`format_audit.md`
- 通过条件：标题、目录、页码、图表题、参考文献、分页均符合规则

#### `/thesis-audit`
- 输入：全文、项目代码、实验数据、联网核验结果
- 输出：`ai_risk_audit.md`、`claim_evidence_matrix.md`
- 通过条件：P0 清零，P1 已修复或显式降级说明

#### `/thesis-reduce`
- 输入：查重高风险段落、原文、事实锚点、引用锚点
- 输出：`reduce_report.md`
- 通过条件：事实、数值、术语、引用未漂移，重复表述显著下降

#### `/thesis-build`
- 输入：已通过审查的正文、图表、引用、格式脚本
- 输出：正式版 docx/pdf、`thesis/release/`
- 通过条件：生成成功，可打开，无目录错乱与资源缺失

#### `/thesis-defense`
- 输入：终稿、实验亮点、系统截图、创新点
- 输出：`defense_outline.md`、`innovation_points.md`、`qa_bank.md`
- 通过条件：能直接支撑答辩讲稿与问答准备

#### `/thesis-sync`
- 输入：最新终稿、图表、文献、答辩材料、notes
- 输出：同步后的 `release/` 与 notes 索引
- 通过条件：正文、图表、文献、终稿版本一致

### Agent Team 提示词铁律

#### `lead`
- 只做总控、分章、统一术语与口径
- 不直接并发改同一章节文件
- 汇总时必须输出：章节状态、证据覆盖、剩余风险

#### `researcher`
- 只查证据，不扩写结论
- 必须回报：代码/日志/图表/文献来源路径
- 对未证实内容必须标 `unverified`

#### `writer`
- 只改分配章节文件
- 不得跨章改标题、图号、引用编号
- 每段关键结论后要能追溯到证据源

#### `citation-checker`
- 只核引用真实性与编号闭环
- 不负责润色正文
- 必须标注文献状态：`verified/partial/unverified/fake-risk`

#### `reviewer`
- findings 优先
- 无问题明确写 `no findings`
- 只审错，不扩写，不重构

### Team 启动模板

```text
角色：<lead/researcher/writer/citation-checker/reviewer>
目标：<本轮唯一目标>
可写文件：<仅列分配文件>
禁止事项：不得扩域；不得修改未分配文件；不得编造证据
交付必须包含：
1. 完成内容
2. 使用的证据源
3. 未证实内容
4. 验证命令或核验动作
```

### Team 收敛清单
- 是否存在多 agent 同写一文件
- 是否有未回报证据源的段落
- 是否有 `fake-risk` 文献未处置
- 是否有图号/表号改动未同步正文
- 是否有结论超出实验支撑

## 最终目标

这个 Skill 的目标不是“快速拼一篇论文”，而是：
- 在要求明确前不乱写
- 在证据不明前不乱断言
- 在文献未核实前不乱引用
- 在排版规则不清前不乱生成
- 在长任务中用 Team 降低上下文污染与目标漂移
- 在查重优化时保住事实锚点与实验结论
